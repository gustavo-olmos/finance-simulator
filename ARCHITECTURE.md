ESPECIFICAÇÃO TÉCNICA: SITE SIMULADOR DE FINANCIAMENTO (MVP)

ESCOPO E OBJETIVO DO AGENTE (CLAUDE) Atue como um Engenheiro de Software Frontend Sênior, especialista em Angular, Arquitetura Serverless e SEO Técnico. O objetivo é implementar um simulador de financiamento imobiliário/automotivo (modelos SAC e PRICE) acoplado a uma listagem de anúncios, rodando 100% Client-Side, sem necessidade de infraestrutura de backend ou banco de dados relacional nesta fase de MVP.

ARQUITETURA DA STACK • Framework: Angular. • Paradigma de Estado: Angular Signals para reatividade síncrona e sem overhead. • Componentização: Standalone Components. • Renderização: Angular SSR (Server-Side Rendering) ativado para indexação de SEO. • Persistência/Dados: Arquivo JSON estático local (src/assets/data/anuncios.json) atuando como Mock de Banco de Dados.

REQUISITOS TÉCNICOS E DIRETRIZES DE IMPLEMENTAÇÃO 3.1. Motor de Cálculo (SAC e PRICE) O agente deve criar um serviço em TypeScript (FinanceCalculatorService). • Inputs: Valor Total do Bem, Valor da Entrada, Taxa de Juros Anual, Prazo (em meses). • Output: Array de objetos contendo: Número da Parcela, Amortização, Juros, Seguro/Taxas e Saldo Devedor. • Regra SAC: Amortização constante. Juros decrescentes recalculados sobre o saldo devedor restante. Parcelas decrescentes. • Regra PRICE: Prestações iguais e constantes. Amortização crescente e juros decrescentes ao longo do tempo.

3.2. Gerenciamento de Estado com Signals • Evitar o uso de RxJS para operações puramente síncronas de formulário. • Utilizar computed signals para derivar o resultado do cálculo (SAC ou PRICE) imediatamente sempre que os inputs do formulário mudarem.

3.3. SEO Técnico (Angular SSR) Para garantir que as páginas de anúncios e a calculadora sejam indexadas de forma eficiente: • Rotas Dinâmicas: /anuncios/:slug para cada item do catálogo. • Serviço de Meta Tags: Implementar um factory injector utilizando os serviços Title e Meta do Angular. • Hydration: Garantir que o estado calculado no servidor seja transferido para o cliente sem oscilação visual (flicker). • Cada página de anúncio deve renderizar no HTML nativo (gerado pelo SSR) as tags og:title, og:description e canonical link preenchidas com os dados do anúncio e os valores simulados padrões baseados no preço do item.

3.4. Camada de Dados Estática (JSON e Tipagem Estrita) Para garantir portabilidade futura para um banco de dados (ex: Supabase/PostgreSQL), o arquivo anuncios.json deve seguir chaves estritas e tipos primitivos equivalentes a colunas SQL. Contrato de Dados (TypeScript Interface) TypeScript export interface Anuncio { id: string; // Equivalente a UUID PRIMARY KEY titulo: string; // Equivalente a VARCHAR/TEXT NOT NULL slug: string; // Equivalente a VARCHAR UNIQUE NOT NULL (Uso em rotas SEO) preco: number; // Equivalente a NUMERIC/DECIMAL NOT NULL descricao: string; // Equivalente a TEXT NOT NULL imagem_url: string; // Equivalente a VARCHAR/TEXT para tags og:image criado_em: string; // Equivalente a TIMESTAMPTZ (Formato ISO 8601) }

INSTRUÇÕES DE CODIFICAÇÃO PARA O CLAUDE (PROMPTS DE EXECUÇÃO) Trabalhe em etapas isoladas seguindo a ordem abaixo: • Etapa 1: Crie o FinanceCalculatorService contendo as funções puras de cálculo para os sistemas SAC e PRICE. Inclua validações para evitar divisão por zero ou prazos negativos. • Etapa 2: Crie o componente de formulário de simulação utilizando os novos componentes standalone e Angular Signals para fazer o binding dos inputs (sliders e campos de texto) integrados ao motor de cálculo. • Etapa 3: Crie um serviço de dados (AnuncioService) que leia o arquivo anuncios.json local.
